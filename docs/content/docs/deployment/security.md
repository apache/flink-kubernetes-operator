# Security

The operator's security surface spans what the operator is allowed to do in the cluster, what its own pods run as, how the operator's and the Flink clusters' network endpoints are protected, which job artifacts it is willing to fetch, and how credentials stay out of the resource specs. Each concern is covered below, with the deeper material linked where it already has a home.

## Access Control

The Helm chart creates two separate service accounts, one for the operator itself and one for the Flink jobs it deploys, each bound to roles with only the permissions its side needs. The scope of these roles follows the installation: cluster-wide by default, narrowed to the watched namespaces when `watchNamespaces` is set. The resources, bindings, and scoping rules are described under [RBAC]({{< ref "docs/deployment/helm/rbac" >}}).

The chart's default roles cover everything the operator does out of the box, with one opt-in exception: the autoscaler's cluster capacity check (`kubernetes.operator.cluster.resource-view.refresh-interval`), described under [Scaling Execution Guards]({{< ref "docs/managing/autoscaler#scaling-execution-guards" >}}), simulates upcoming scale-ups against the cluster's free resources and needs three reads beyond the defaults. Listing the nodes is covered by the chart's optional `rbac.nodesRule.create`, while node usage (the `metrics.k8s.io` API) and the Cluster Autoscaler status ConfigMap in `kube-system` need rules added by the installation. Missing permissions never break scaling: the check disables itself with a warning in the operator log.

## Pod Security

The operator pod runs as a non-root user by default, the chart's `podSecurityContext` sets `runAsUser: 9999` and `runAsGroup: 9999`. Further hardening, such as read-only root filesystems or dropped capabilities, is applied through the `operatorSecurityContext` and `webhookSecurityContext` values, listed with the other parameters under [Installation]({{< ref "docs/deployment/helm/installation" >}}). The Flink cluster pods carry their own security settings, defined per resource through the [Pod Template]({{< ref "docs/custom-resource/pod-template" >}}).

## Webhook TLS

The admission webhook serves over TLS, with cert-manager issuing its certificate: the chart creates a self-signed issuer and a certificate for the webhook service, delivered as a PKCS12 keystore to the webhook container. Two values shape the keystore:

- `webhook.keystore.passwordSecretRef`: the secret holding the keystore password, defaulting to a secret generated by the chart.
- `webhook.keystore.pkcs12Profile`: the keystore encryption profile, for FIPS-compliant environments.

The full setup, and what disabling the webhooks removes, is described under [Cert Manager]({{< ref "docs/deployment/helm/cert-manager" >}}).

## Cluster Connection TLS

The operator talks to the Flink clusters it manages through their REST endpoints, and these connections can run over TLS. With `tls.create: true`, the chart mounts the secret named by `tls.secretName` at `/opt/flink/tls-cert`, and the operator picks up what it finds there:

- A `truststore.jks` enables server verification: the operator sets it as the REST truststore for every cluster connection.
- A `keystore.jks` additionally enables mutual TLS: when a job's configuration turns on `security.ssl.rest.authentication-enabled`, the operator presents the client certificate from the keystore.

The password for both stores is read from the secret referenced by `tls.secretKeyRef`. Setting up the certificates on the Flink side is standard Flink configuration, covered by the Flink [SSL Setup](https://nightlies.apache.org/flink/flink-docs-master/docs/deployment/security/security-ssl/) documentation.

## Flink Web UI Exposure

By default the Flink REST endpoint, which also serves the Web UI, is only reachable inside the cluster. Exposing it externally is an explicit choice, either through Flink's [service expose types](https://nightlies.apache.org/flink/flink-docs-master/docs/deployment/resource-providers/native_kubernetes/#accessing-flinks-web-ui) (`kubernetes.rest-service.exposed.type`) or through the Ingress entries the operator generates per resource. The Ingress path carries standard Kubernetes TLS through `spec.ingress.tls`, covered under [Ingress]({{< ref "docs/custom-resource/ingress" >}}).

## Artifact Fetching

For a `FlinkSessionJob`, the operator itself downloads the job artifact referenced by `spec.job.jarURI` before submitting it to the session cluster, so the URI is validated up front and the fetch itself is configurable:

- `kubernetes.operator.user.artifacts.allowed-schemes` (default `https`): the allowlist of URI schemes. Installations fetching artifacts from other stores, for example `s3` or `hdfs`, extend this list explicitly and add the matching Flink filesystem plugin to the operator, as described under [Additional Dependencies]({{< ref "docs/deployment/plugins#additional-dependencies" >}}).
- `kubernetes.operator.user.artifacts.disallow-restricted-hosts` (default `true`): rejects `http` and `https` URIs whose host resolves to a loopback, link-local, site-local, wildcard, or multicast address, so a session job cannot point the operator at cluster-internal endpoints.
- `kubernetes.operator.user.artifacts.http.header`: custom HTTP headers sent when fetching artifacts over `http` and `https`, typically carrying the credentials of the artifact store.

Application-mode deployments are not affected: their `jarURI` is resolved inside the job's own cluster, not by the operator.

## Secrets in Configuration

Values in `spec.flinkConfiguration` are stored in plaintext: they are readable by anyone allowed to read the custom resource, and the operator additionally persists the last reconciled spec in the resource status. Credentials therefore do not belong in the configuration directly. The established pattern is to keep them in Kubernetes Secrets and inject them through the [Pod Template]({{< ref "docs/custom-resource/pod-template" >}}), as environment variables or mounted files, so the resource carries only the reference.

Configuration values also reach the operator logs. Every spec change is audit-logged together with its diff, so a value placed in `spec.flinkConfiguration` becomes readable for anyone with access to the logs, an audience the RBAC rules on the resource itself do not control. Injecting credentials at runtime keeps them out of this path as well.
